# 시연 직전 화면 문제와 절대 좌표 우회

- 발생 시점: UNITHON 2024 시연
- 기록일: 2026-07-15
- 범위: 키오스크 메뉴 자동 조작
- 상태: 구조적 원인 개선, 실제 Windows 키오스크 재검증 필요

## 맥락

해커톤 시연 직전 키오스크 화면 동작에 문제가 생겼고, 제한된 시간 안에 시연 흐름을 끝내기 위해 메뉴와 버튼의 좌표를 코드와 JSON에 직접 넣었다. 이 우회는 당시 정해진 화면에서 흐름을 진행하기 위한 대응이었다.

## 기대 동작과 실제 동작

기대 동작은 음성 주문에 맞는 카테고리·페이지·메뉴를 찾아 장바구니에 담는 것이었다. 실제 클라이언트는 1080×1920 기준의 특정 영역과 2×8 카드 그리드를 가정하고 저장 좌표를 클릭했다. 클릭 API에 예외가 없으면 카테고리 이동, 페이지 이동, 항목 추가가 성공한 것으로 내부 상태를 갱신했다.

당시 문제를 촉발한 정확한 UI 변경, 배율, focus 상태와 로그는 보존돼 있지 않다. 따라서 특정 픽셀 오차를 사후에 단정하지 않는다.

## 영향

- 다른 해상도·DPI·창 위치에서 좌표가 빗나갈 수 있었다.
- 메뉴 순서나 페이지 구성이 바뀌면 오래된 메뉴를 선택할 수 있었다.
- 옵션 모달이 열리거나 클릭이 무시돼도 장바구니 추가로 기록할 수 있었다.
- 결제 화면 도달 여부를 확인하지 못했다.
- 음성 backend가 보낸 온도·사이즈 의미를 잃었다.
- 주문 전달 중 프로세스가 종료되면 유실·중복 여부를 판단할 수 없었다.

## 재현 가능한 구조적 결함

정확한 시연 장애는 재현 자료가 없지만 기존 코드에서 다음 구조는 확인할 수 있었다.

1. `ocrFirst.py`와 `kioskMenuCapture.py`가 기준 화면 영역을 고정했다.
2. 메뉴 인덱스가 `(category, page, absolute_xy)`만 보존했다.
3. `Navigator.click()`의 포인터 호출 성공이 UI 결과 성공으로 전파됐다.
4. 실행 중 화면 재관찰과 장바구니 수량 비교가 없었다.
5. 주문 허브가 새 POST로 이전 주문 한 칸을 덮어쓸 수 있었다.

## 가설과 판단

검토한 촉발 가설은 브라우저 배율·DPI, 창 크기·위치, 렌더링 지연, focus, UI 레이아웃 변경이다. 보존된 증거만으로 어느 하나를 확정할 수 없다. 그러나 어떤 촉발 조건이었더라도 결과 검증 없는 절대 좌표가 실패를 감지·복구하지 못한 구조적 원인이라는 점은 코드에서 확인된다.

## 해결

- Live 대상 창 제목과 handle 고정
- UI Automation 의미 탐색과 Invoke 우선
- UIA 목표 누락 시 대상 창 EasyOCR fallback
- 좌표 fallback 기본 비활성화와 창 상대 스케일링
- 의미 상태 전이와 두 번 연속 postcondition 안정화
- 메뉴 영역의 실제 페이지 증거 확인
- 항목 추가 전후 장바구니 의미 delta 확인
- backend `displayName`, 온도, 크기 보존
- SQLite single-claim, 멱등 키, 결과 ACK, `awaiting_handoff`·`uncertain` 수동 복구
- 정확한 native window handle 고정과 offscreen·disabled UIA control 제외
- Live 주문 허브의 설치별 token 인증
- 결제 방법 선택 화면에서 정지하고 결제 확인 금지

## 검증

표준 라이브러리 테스트로 다음 실패를 재생했다.

- 같은 이름의 두 버튼이 있을 때 모호성 거부
- 클릭 API 성공 후 화면이 바뀌지 않은 경우 실패
- 애니메이션 hash만 바뀌고 장바구니 수량이 그대로인 경우 실패
- 장바구니에만 목표 페이지 메뉴명이 보이는 경우 페이지 이동 실패
- `카드`가 `카드를 넣어주세요` terminal로 오인되지 않음
- 같은 DB에서 두 worker가 동시에 주문을 claim하지 못함
- action 결과 불확실 시 다음 주문 차단
- 검증된 live 장바구니 변경 후에도 고객 인계·초기화 전 다음 주문 차단
- OCR 확대 polygon의 원본 좌표 복원

실제 Windows UIA, OCR 모델, 해커톤 당시 장치와 물리 포인터 end-to-end는 현재 macOS 환경에서 검증하지 않았다. 대상 Windows 키오스크 acceptance가 남은 검증 gate다.

## 회귀 방지

- 성공 상태에는 관찰된 postcondition이 반드시 있어야 한다.
- 장바구니 변경은 일반 화면 hash가 아니라 지정 영역의 의미 텍스트 delta로 확인한다.
- action 이후 불확실성을 일반 실패로 낮추거나 자동 재시도하지 않는다.
- 새 키오스크는 좌표 복사보다 read-only 진단과 프로필 review를 먼저 수행한다.
- 실제 장애와 검증 결과가 생길 때만 이 기록에 추가하며 지표를 추정하지 않는다.

## 인터뷰에서 설명할 질문

- 왜 YOLO 전체 학습보다 UIA + OCR를 선택했는가?
- 분산 exactly-once가 불가능한 물리 UI에서 중복 주문을 어떻게 막았는가?
- 화면 변화와 실제 장바구니 반영을 어떻게 구분했는가?
- 좌표 fallback이 필요한 경우에도 어떤 안전 불변 조건을 유지하는가?
- 자동화의 접근성 가치와 법적 적합성 주장을 어떻게 분리했는가?
